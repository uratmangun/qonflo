## Context

The existing `scripts/skills-update-harnesses.sh` requires `skills-lock.json` and exits when it is missing. The workspace already has installed skills in `.agents/skills`, and `npx skills list --json` can enumerate their names and paths, but it does not expose original install sources.

The current update harness expects every lock entry to be directly installable with `npx skills add <source> -s <skill>`. Some installed skills fit that model, while others do not. OpenSpec skills are generated by `openspec update`, and `pencil-design` is copied from the `@pencil.dev/cli` npm package or CDN raw `SKILL.md` rather than installed through a normal skills.sh source.

## Goals / Non-Goals

**Goals:**
- Provide a script that bootstraps `skills-lock.json` when missing.
- Provide an explicit existing-lock flow that can edit selected source metadata or quit without changes.
- Generate a lock that is safe for the current update harness to consume.
- Avoid inventing unknown sources from local skill names alone.
- Make skipped/non-standard skills visible so they can be handled by a future provider-aware update flow.

**Non-Goals:**
- Do not replace `npx skills update`.
- Do not automatically overwrite or regenerate existing `skills-lock.json` files.
- Do not implement provider-aware updates for OpenSpec, raw `SKILL.md` URLs, npm package skills, or `node_modules` skills in this change.
- Do not attempt network registry searches during generation; source mappings should be deterministic and reviewed in the script.

## Decisions

### Add a Dedicated Generator Script
Create a new script, for example `scripts/skills-create-lock.sh`, instead of folding lock generation into the update harness immediately.

Rationale: lock creation and skill updating have different risk profiles. Keeping the generator separate prevents update runs from unexpectedly writing a new lock.

Alternative considered: make `skills-update-harnesses.sh` auto-generate when missing. This is convenient but makes an update command mutate repository metadata before the user has reviewed generated sources.

### Offer Edit or Quit When Lock Exists
When `skills-lock.json` already exists, the script should not exit immediately. It should present an explicit choice: edit an existing skill entry or quit without changes. Editing should be limited to fields that affect update source resolution: `source`, `sourceType`, and `skillPath`.

Rationale: users may need to correct or manually add provenance after a lock has been generated, especially for skills installed manually or from ambiguous registry results.

Alternative considered: exit successfully whenever the lock exists. This protects the file but forces manual JSON editing for common source corrections.

### Validate Supported Source Types
The current `skills` CLI source parser produces these normal source types for `npx skills add` inputs:

| Source Type | Supported Input Examples |
|---|---|
| `github` | `owner/repo`, `github:owner/repo`, `https://github.com/owner/repo`, GitHub tree URLs with optional subpaths |
| `gitlab` | `gitlab:group/repo`, `https://gitlab.com/group/repo`, GitLab tree URLs with optional subpaths |
| `git` | SSH or generic git URLs not classified as GitHub/GitLab |
| `local` | Local filesystem paths |
| `well-known` | HTTP(S) hosts serving `/.well-known/agent-skills/index.json` or `/.well-known/skills/index.json` |

The CLI also writes optional lock metadata such as `ref`, `skillPath`, and `computedHash`. For project updates, `skillPath` is important because `npx skills update -p` treats entries without `skillPath` as legacy/non-updatable in place.

Rationale: validating `sourceType` prevents typos like `github-url` or provider-specific values that the current harness cannot use.

Alternative considered: accept any string for `sourceType`. This makes the lock more flexible but creates failures in the existing update harness.

### Prefer `npx skills list --json`, Fallback to `.agents/skills`
Use the CLI inventory when available because it reflects project skills across configured agents and canonical paths. If that fails or returns no project skills, scan `.agents/skills/*/SKILL.md` and parse `name:` from YAML frontmatter.

Rationale: the CLI gives a normalized installed inventory, while direct scanning keeps the script useful in partial or broken CLI states.

Alternative considered: only scan `.agents/skills`. This misses cases where the CLI can see installed project skills through other configured harness directories.

### Use a Static Known Source Map
Embed a conservative mapping for currently known compatible skills:

| Skill | Source | Source Type |
|---|---|---|
| `cloudflare` | `cloudflare/skills` | `github` |
| `gh-cli` | `github/awesome-copilot` | `github` |
| `hono` | `yusukebe/hono-skill` | `github` |
| `mcporter` | `steipete/clawdis` | `github` |
| `react-devtools` | `callstackincubator/agent-react-devtools` | `github` |
| `skill-creator` | `anthropics/skills` | `github` |
| `webauthx` | `wevm/webauthx` | `github` |

Rationale: these sources were identified through prior registry/source exploration and are compatible with the current harness command shape.

Alternative considered: query `npx skills find` inside the generator. This is not deterministic enough for a lock file because search can return multiple matches and has no stable JSON contract in the current CLI help.

### Skip Non-Compatible Skills for Current Harness Compatibility
The generator should not include `pencil-design` or `openspec-*` in the generated lock until the update harness supports provider-specific source types. It should report them as skipped with reasons.

Rationale: adding non-compatible entries would make the current update harness attempt invalid `npx skills add` commands. The script already continues on failure, but generating known-failing entries creates noisy and misleading behavior.

Alternative considered: include skipped entries with `skipUpdate: true`. The current harness ignores `skipUpdate`, so this would not be safe until the harness is changed.

## Risks / Trade-offs

- Static mappings can become stale or incorrect → Keep mappings small, explicit, and easy to review in the script.
- Some installed skills may have been customized locally → The generated lock updates from upstream sources and may overwrite local customizations when the update harness runs.
- Skipping non-standard skills means the lock is incomplete → Print clear skipped output and reserve provider-aware entries for a later change.
- Parsing YAML frontmatter with shell tools can be brittle → Prefer `npx skills list --json`; fallback parsing should only extract simple `name:` lines.
- Interactive editing can corrupt JSON if implemented with string replacement → Use `jq` to apply all field updates.

## Migration Plan

1. Add the generator script.
2. Run it in repositories missing `skills-lock.json`.
3. Review the generated lock before running `scripts/skills-update-harnesses.sh`.
4. Use the existing-lock edit flow to correct `source`, `sourceType`, or `skillPath` when needed.
5. Handle skipped skills manually or with a future provider-aware harness change.

Rollback is deleting the generated `skills-lock.json` if it was not wanted.

## Open Questions

- Should source mappings eventually move to a separate data file for easier review?
- Should provider-aware updates for OpenSpec and Pencil be added to the same harness or a separate script?
